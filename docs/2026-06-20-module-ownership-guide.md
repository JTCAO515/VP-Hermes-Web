# VisePanda 模块责任建议表

## 1. 文档目的

这份文档是写给接手的 coding agent 的模块视角说明。  
它的目标不是替代文件树，也不是组织架构图，而是帮助接手者理解：

- 这个项目当前最适合按哪些“责任中心”来理解
- 每个模块最重要的角色是什么
- 它与其他模块的耦合点在哪里
- 改这个模块时最容易误伤哪里

如果没有这份视角，接手者很容易只按文件来改，结果是：

- 改 `Home` 顺手碰了 `Chat`
- 改 `Tools` 顺手碰了 `Trips`
- 改 `Shared Infra` 却低估了它对全站的影响

## 2. 背景

这个项目的模块边界，不是从一开始就按现代前端框架、组件系统或后端微服务边界切好的。  
它更像是围绕产品主链路，在持续迭代中长出来的几个责任中心：

- 首页要承担第一印象和入口作用
- Chat 是真正的主链路入口
- Cities 和 Tools 是结构化知识层
- Trips 是保存与回访层
- Admin 是内部管理层
- Shared Infra 是所有页面共用的壳层、导航、状态与配置

这意味着当前最适合的理解方式不是“理想分层”，而是：

> **先按模块责任来理解，再按文件定位实现。**

这样做对下一个 coding agent 更友好，因为它会先知道“自己到底在改哪一块业务角色”，而不是一头扎进文件细节里。

## 3. 模块责任表

## 3.1 Home

### 当前角色

`Home` 是站点的第一印象层，也是用户决定“这个网站是不是在帮我解决中国旅行问题”的第一入口。

### 为什么重要

它负责把项目的价值说清楚：

- 这不是通用 AI 助手
- 这是中国旅行产品站
- 这里的内容、工具和聊天是互相关联的

### 与哪些模块耦合最强

- `Chat`
- `Cities`
- shared navigation

### 当前最值得处理的问题

- 首页到主功能入口之间的转化是否自然
- 首屏体验是否稳
- 入口动作是否足够清晰

### 当前最不该顺手处理的问题

- 不要因为改首页就顺手重构 auth
- 不要因为首页排版想法改变就连带重写整个 navigation 结构

## 3.2 Chat

### 当前角色

`Chat` 是项目真正的主链路入口，也是最接近“产品核心价值”的部分。

### 为什么重要

对用户来说，这个项目最终是不是有价值，很大程度取决于：

- Chat 是否能正常触发
- 流式响应是否顺畅
- 回复是否像旅行规划，而不是泛聊

### 与哪些模块耦合最强

- auth
- SSE 输出
- trips
- prompt / knowledge 注入

### 当前最值得处理的问题

- 线上稳定性
- 输入到回复的反馈感
- Chat 输出与 Cities / Tools / Trips 的联动

### 当前最不该顺手处理的问题

- 不要在无明确问题时重写聊天协议
- 不要顺手改动与 auth / history 紧耦合的数据结构

## 3.3 Cities

### 当前角色

`Cities` 是项目的结构化知识层，负责把“AI 口头建议”变成可浏览、可比较、可沉淀的城市信息。

### 为什么重要

它补足了 Chat 的一个天然短板：  
聊天是流动的，城市页是稳定的。  
用户需要一个地方，在不反复提问的情况下浏览目的地信息。

### 与哪些模块耦合最强

- map
- tools
- 图片资源
- 数据层

### 当前最值得处理的问题

- 城市详情信息的清晰度
- 城市页到工具或聊天的下一步动作
- 数据覆盖不均的问题

### 当前最不该顺手处理的问题

- 不要因为城市卡片想优化就顺手重做地图层
- 不要轻易调整数据结构，只因为当前表达不够理想

## 3.4 Trips

### 当前角色

`Trips` 是用户保存与回访层。  
它承接的是“用户决定把这次规划留下来”的动作。

### 为什么重要

如果没有 `Trips`，这个项目更像一次性的聊天站。  
有了 `Trips`，它才更像一个可持续使用的旅行产品。

### 与哪些模块耦合最强

- auth
- chat history
- timeline
- tools

### 当前最值得处理的问题

- `Trips → Tools` 的联动
- 已保存内容的可读性
- 回访体验是否自然

### 当前最不该顺手处理的问题

- 不要为了改 trip 页面而轻易改动 auth 主链路
- 不要在没有清楚目标时重做 persistence 模型

## 3.5 Tools

### 当前角色

`Tools` 是旅前辅助工作台。  
它提供的是用户在做决策时真正会反复查看的信息，例如签证、预算、短语、急救、打包。

### 为什么重要

它让项目不只是“会聊天”，而是开始变成“有工具价值的产品”。

### 与哪些模块耦合最强

- `data/tools.json`
- tool detail overlay
- Cities
- Trips

### 当前最值得处理的问题

- 工具与主链路之间的联动
- 工具内容深度
- 工具的使用场景是否足够明确

### 当前最不该顺手处理的问题

- 不要因为补工具内容就顺手大改 trip persistence
- 不要把工具页演变成新的信息架构重构入口

## 3.6 Admin

### 当前角色

`Admin` 是内部管理和状态查看层，不是用户主链路的一部分。

### 为什么重要

它决定项目是否有基本的运营与管理抓手：

- 用户
- 聊天记录
- 基础统计

### 与哪些模块耦合最强

- auth
- admin routes
- chat logs

### 当前最值得处理的问题

- 基本可用性
- 权限边界
- 信息是否足够支持排查和管理

### 当前最不该顺手处理的问题

- 不要因为 admin 不够漂亮就优先大改它
- 不要让 admin 的调整反向影响前台主链路

## 3.7 Shared Infra

### 当前角色

`Shared Infra` 不是一个单独页面，而是全站共同依赖的基础层，包括：

- bootstrap
- navigation
- shared state
- version/config
- 共用的 overlay / modal / loading / error 逻辑

### 为什么重要

它虽然不显眼，但一旦出问题，往往会同时影响多个页面。  
`v5.0.9` 的稳定性专项，本质上就是对这层做了集中修补。

### 与哪些模块耦合最强

- 全站

### 当前最值得处理的问题

- 稳定性
- 初始化顺序
- 公共反馈状态
- 跨页面行为一致性

### 当前最不该顺手处理的问题

- 不要在没有测试和明确目标时大改公共 bootstrap
- 不要把“顺手整理”变成“全站公共层重写”

## 4. 多 agent / 多人协作时的边界建议

如果未来不是一个单一接手者，而是多个 agent 或多人协作，这一节尤其重要。

### 改 Home 时

- 默认只动首页展示与入口
- 不要顺手改 auth 主链路
- 不要顺手改全局 navigation 契约

### 改 Chat 时

- 先确认 SSE 输出与前端消费格式的契约
- 不要顺手改历史记录模型
- 不要在无必要时同时改 prompt、auth、trip persistence

### 改 Cities 时

- 先确认是否会影响 map 和图片链路
- 不要把数据问题顺手升级成整层数据结构重做

### 改 Trips 时

- 先确认 auth / profile / history 是否会被牵连
- 不要因为页面体验优化就轻易重构保存模型

### 改 Tools 时

- 先判断当前任务是“内容增强”还是“主链路联动”
- 不要因为改工具页而顺手碰 trip persistence

### 改 Admin 时

- 保持 admin 问题局部化
- 不要让后台改动反向污染前台体验

### 改 Shared Infra 时

- 默认把它视为高风险改动
- 任何公共层变化都要假设会影响全站
- 在没有明确收益时，不要把 shared 层当作重构练习场

## 5. 给下一个 coding agent 的一句话建议

> **先决定你在改哪个模块，再决定你要打开哪些文件；不要因为文件近，就顺手把别的模块一起改了。**
