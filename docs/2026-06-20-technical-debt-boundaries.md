# VisePanda 技术债边界说明

## 1. 文档目的

这份文档不是要列出项目里所有不完美的地方，而是帮助接手的 coding agent 判断：

- 哪些技术债当前先接受
- 哪些债现在不要主动碰
- 哪些债可以在当前任务里顺手修
- 哪些债只有在触发条件成立时，才值得升级成主任务

这份文档存在的原因很现实：  
新接手的 agent 很容易把“看见不完美”误判成“必须先重构”。  
但这个项目当前更需要的是稳步推进，而不是把注意力耗在提前发生的大手术上。

## 2. 背景

这个项目的结构不是从完美的分层设计里长出来的，而是从连续迭代里长出来的。

它经历过几次明显的阶段变化：

- 先把中国旅行 AI 规划做出来
- 再补 Auth、Trips、Admin、Visa
- 再做英文原生化
- 再做 Tools、移动端和稳定性

因此你今天看到的很多“债”，其实是当时阶段下的合理取舍。  
例如：

- `web/app.js` 为什么会很重  
  因为前端主逻辑是在持续功能迭代里逐渐集中起来的，项目优先级一直是“先把主链路跑通”

- 为什么 `static/*` 兼容层还保留  
  因为项目从更早的脚本组织方式演进到现在的主 SPA 壳层，旧层没有全部清空，但也并不等于它们现在都是主路径

- 为什么还继续保留 `Vercel`  
  因为当前更需要的是稳定迭代和产品验证，而不是立刻承担更大的部署和运维复杂度

所以，“存在技术债”并不自动等于“现在就要解决”。  
接手时最重要的不是技术洁癖，而是判断时机。

## 3. 当前应接受的技术债

这类问题当前确实存在，但在现阶段是可以接受的。

### `web/app.js` 仍然偏大

这是当前最明显的工程债之一。  
但它虽然偏重，仍然承载着站点的主要前端逻辑，且最近几轮稳定性和工具页工作都是在这个基础上继续推进的。

当前判断：

- 它确实不够理想
- 但它还没有糟糕到“必须先拆才能继续做事”
- 继续在这个文件上做小范围、有边界的改动，仍然是当前可接受的方式

### 数据层存在并行形态

项目里同时存在：

- `data/*.json`
- `data/knowledge/*.py`

这会让第一次接手的人觉得结构不统一。  
但这并不等于当前一定有逻辑错误，它更多是阶段性演化的痕迹。

当前判断：

- 先理解哪条链路当前活跃
- 不要因为“看起来不统一”就先动数据结构

### 兼容层仍保留

`static/*` 和部分历史脚本层还在仓库里。  
这会让人有一种“应该先清理”的冲动。

当前判断：

- 先识别是否活跃
- 未确认前，不应默认删除

### 当前仍保留 `Vercel`

`Vercel` 有天然边界，例如冷启动、流式接口可控性、资源治理能力。  
但当前阶段继续保留它，仍然是合理选择。

当前判断：

- 部署边界是真实存在的
- 但当前不需要因为局部问题立刻整站迁移

## 4. 当前不应主动触碰的技术债

这部分是最重要的边界说明。

### 不要无明确目标地大拆 `web/app.js`

如果没有一个明确的当前任务，例如：

- 某块逻辑已经反复阻碍开发
- 某块逻辑已明确导致线上 bug
- 某块逻辑已导致测试无法维护

那么现在不该主动把 `web/app.js` 大规模拆开。  
因为这会带来三种风险：

1. 改动面过大
2. 影响全站多个 view
3. 让当前正在收口的产品节奏被打断

### 不要无明确收益地大清理 `static/*`

兼容层的存在确实不优雅，但“先把旧东西清掉”通常不是当前最有价值的工作。  
如果没有非常明确的证据证明：

- 它正在影响主路径
- 它正在误导开发
- 它正在制造线上风险

那么不要优先花大量时间清理。

### 不要因为局部问题立刻整站迁移部署

当前已经有明确的商用升级路线文档。  
但那份路线的核心不是“今天开始迁”，而是“当前继续保留，未来按条件迁”。

如果只是出现：

- 某个页面响应慢
- 某张图加载失败
- 某个按钮有问题

这些都不足以支持“立刻整站迁移”的决策。

### 不要在无明确目标下重构 auth / chat / trips 主链路

这些链路之所以危险，不是因为它们写得最好，而是因为它们已经承载真实业务价值：

- 用户登录
- 行程保存
- 聊天历史
- 管理后台

如果没有一个明确的用户问题或业务目标，不要主动把这类链路拉出来重写。

## 5. 可以顺手修的小债

并不是所有技术债都要被压着不动。  
下面这些问题，通常适合在当前任务里顺手修：

### 小范围样式问题

例如：

- 边距异常
- 文案折行
- 移动端局部遮挡
- 轻量级的安全区补丁

### 独立文案或说明错误

这类问题风险低，且对最终体验直接有帮助。

### 工具数据和展示层的小修

例如：

- `tools.json` 某条文案不自然
- 工具详情里某段标题表达不够清晰
- 某个 fallback 提示不够自然

### 局部 fallback / error / empty state

这类小债通常非常值得顺手修，因为它们能明显减少“点了像没反应”的体验问题。

## 6. 什么时候技术债才值得升级为主任务

不是所有债都值得升格成主任务。  
下面这些情况成立时，才值得考虑把技术债单独提级处理。

### 情况 1：当前任务反复被同一结构问题阻塞

如果你连续几次都发现：

- 每次改某块都绕不开同一坨结构
- 每次修一个小问题都会影响好几处
- 同一段代码已经明显成为开发瓶颈

那么这类债就不再只是“难看”，而是“阻碍推进”。

### 情况 2：线上问题已明确归因到该技术债

例如：

- 初始化链路太集中导致公共故障
- 资源加载逻辑分散导致线上反复缺图
- 兼容层与主链路混用导致行为不一致

这时它已经不是工程洁癖问题，而是产品质量问题。

### 情况 3：多人或多 agent 并行增加

当多个 agent 同时改同一块逻辑时，之前单人还能扛住的混乱结构，会迅速变成协作风险。  
这时某些技术债就会从“可忍受”升级成“必须处理的边界问题”。

### 情况 4：商用要求开始逼近结构上限

如果后续真的进入更强的商用品质要求，例如：

- 更稳的后端服务
- 更明确的日志与监控
- 更清晰的资源治理
- 更强的认证和权限控制

那么部分“现在先接受”的债，会自然升级为主任务。

## 7. 当前的技术债判断原则

如果你要快速判断某个债要不要现在动，可以用这 4 个问题：

1. 它是不是正在阻碍当前任务推进  
2. 它是不是已经造成线上问题  
3. 它是不是已经让多人协作变危险  
4. 它是不是已经直接影响商用准备

如果四个问题都是否定的，那么大概率现在还不该把它升格成主任务。

## 8. 给下一个 coding agent 的一句话建议

> **不要把“看见技术债”误解成“现在就必须先还债”；先判断它是不是已经阻碍产品推进。**
