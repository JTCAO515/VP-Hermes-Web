# VisePanda 未来 2-4 周推荐迭代顺序

## 1. 文档目的

这份文档是写给下一个 coding agent 的短中期推进说明。  
它不是长期 roadmap，也不是产品愿景文档，而是帮助接手者在未来 2-4 周里判断：

- 什么事情现在最值得先做
- 什么事情看起来重要，但其实应该排后
- 为什么当前不建议把精力放在大重构或整站迁移上

如果只看代码结构，很容易误以为当前最该做的是“把结构整理漂亮”。  
但从项目演进历史来看，当前更重要的是在已有资产上继续收口、补全和稳定，而不是为了架构纯度把节奏打散。

## 2. 背景

这个项目最初不是从一套完整架构设计长出来的，而是从一个垂直场景原型一步步迭代出来的。

早期阶段，重点是先把几个关键问题做出来：

- 能不能用 AI 规划中国旅行
- 能不能把中国城市知识组织成结构化内容
- 能不能让用户保存 trip、登录、回来看历史

等这些主能力逐步成型以后，项目又继续补了几层：

- Auth、Trips、Admin
- 英文原生化
- Tools 详情页
- 移动端优化
- `v5.0.9` 的稳定性专项修复

因此，今天看到的项目不是“设计完美但功能不全”的系统，而是“功能已经长出来、工程边界逐步补齐”的系统。  
这个背景很重要，因为它决定了接下来最合理的工作方式：

> **优先继续提升产品完成度和稳定性，而不是先把所有结构整理成理想形态。**

## 3. 当前判断

当前项目所处阶段，可以概括为：

1. 已经不是 demo  
2. 但还没有达到成熟商用产品  
3. 当前最值钱的是已经完成的产品资产，而不是代码结构的“理论最优”  
4. 接下来 2-4 周最有价值的工作，是继续提升主链路质量和产品闭环

从用户角度看，当前最重要的是：

- 主站是否稳定
- 核心页面是否顺
- 功能之间是否形成闭环
- 网站是否越来越像一个真正可用的产品，而不是一组页面

从接手 agent 的角度看，当前最重要的是：

- 不要同时开太多线
- 不要因为发现技术债就先重构
- 不要让“修一个点”膨胀成“顺手重写一层”

## 4. 推荐推进顺序

下面的顺序不是唯一真理，但在当前项目状态下，是最稳也最划算的顺序。

### 第一优先级：继续做线上稳定性复测与小修

这一层工作应该持续存在，不要因为 `v5.0.9` 已经做过稳定性专项，就默认问题已经完全收口。

优先关注：

- `Sign in`
- 图片加载与 fallback
- Chat / Cities / Trips / Tools 切换反馈
- 手机端 bottom nav 与布局
- 多浏览器一致性

推荐动作：

- 先按已有回归手册做线上复测
- 把真实问题记录为一小批、一小批修
- 每次只修用户能直接感知的问题，不借机扩大范围

### 第二优先级：补产品闭环，而不是继续堆散点功能

当前站点的页面已经不少，但页面之间的联动仍不够强。  
最值得优先补的不是再加新页面，而是把已有页面连起来。

推荐顺序：

1. `Trips → Tools`
2. `Cities → Tools → Chat`
3. `Cities / Tools / Chat / Trips` 的状态与内容衔接

推荐动作：

- 让 trip 页面能更自然地带出 packing / pricing / phrases / visa
- 让城市页不是“看完就关”，而是能把用户带去下一步动作
- 让 Chat 的输出和 Tools / Cities 的结构内容互相增强

### 第三优先级：补商用品质，而不是立刻做大迁移

当前商用方向已经明确，但现在的工作重点仍然不是整站迁移，而是为未来迁移做准备。

推荐动作：

- 继续梳理哪些 API 最适合后续独立出去
- 记录资源、数据库、日志与监控的缺口
- 在不改部署形态的前提下，先把工程边界看清楚

这一阶段更像“为商用做准备”，而不是“今天就开始迁移”。

## 5. 为什么是这个顺序

### 先稳住体验，因为这决定了项目值不值得继续投入

如果线上体验仍然不稳，那么再漂亮的中台化、分层化和迁移计划，短期都不会给用户带来明显收益。  
而稳定性、小修和主链路闭环，是用户最能直接感受到的改进。

### 先补闭环，因为当前的核心问题不是页面不够多，而是页面之间还不够连

这个项目已经有 Home、Chat、Cities、Trips、Tools、Admin。  
下一步真正提升“产品完成度”的方式，不是再加新 tab，而是让已有 tab 之间形成更自然的流动。

### 把商用准备排在第三位，因为现在最合理的是“准备”，不是“迁移”

商用方向当然重要，但如果现在就把大量时间投到部署切换、服务拆分和结构重整，反而会让产品主线停下来。  
因此当前更合理的做法，是：

- 先确认最核心的业务链路稳定
- 再把未来独立后端需要的信息补齐
- 最后在时机成熟时推进迁移

## 6. 偏离顺序的触发条件

这份顺序不是一成不变的。以下情况出现时，优先级应该调整。

### 情况 1：线上再次集中暴露质量问题

如果你在真实站点上连续发现：

- 登录问题
- 图片失效
- 切换卡死
- 移动端导航再次消失

那么优先级应立即回到“稳定性优先”，暂停新功能联动工作。

### 情况 2：开始真正推进商用准备

如果业务已经明确进入商用阶段，例如开始认真准备：

- 支付
- 订单
- 更正式的用户运营
- 长期可持续的服务可控性

那么“后端独立化”和“资源治理”优先级应上升。

### 情况 3：接手者从单一 agent 变成多 agent 或多人协作

如果未来不是一个主接手 agent，而是多个 agent / 多个开发者并行，那么优先级应提高：

- 模块责任边界
- 高风险文件修改约束
- shared infra 的变更纪律

因为并行协作时，边界混乱比单点技术债更危险。

## 7. 当前不建议优先做的事

这部分不是说永远不做，而是说“在未来 2-4 周不建议优先做”。

- 不建议先大拆 `web/app.js`
- 不建议先大清理 `static/*`
- 不建议先做整站迁移
- 不建议先改 auth / chat / trips 数据主链路
- 不建议为了看起来更现代而先换技术栈

这些事情一旦做得太早，很容易把当前最值钱的东西打断：  
也就是已经逐步成型的用户主链路和产品完成度。

## 8. 给下一个 coding agent 的一句话建议

如果你只能记住一句话，就记住这个：

> **先把这个项目继续做成“更完整、更稳定的产品”，再去把它做成“更漂亮的系统”。**
